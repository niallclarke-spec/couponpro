1. Platform Overview

PromoStack is a multi-tenant SaaS marketing automation platform with two primary products delivered under the EntryLab brand:

  • PromoStack Coupon Bot — Telegram bot for branded promotional image generation, used by marketing teams to create coupons at speed.
  • EntryLab Forex Signals — Automated XAU/USD (gold) trading signals with AI-powered narrative messages, delivered to paying VIP subscribers via Telegram.

The platform also ships a suite of supporting sub-systems: a Journeys / Support Chat engine (multi-step Telegram conversational flows), a Hype Bot (AI-powered free-channel marketing automation), a Cross Promo worker (automated drip of VIP teasers into the free channel), and a Conversion Tracking pipeline (email → free Telegram invite → paid VIP).

4. Authentication & Access Control

Primary Auth — Clerk JWT

  • Token carried in Authorization: Bearer <token> header or __session cookie.
  • JWKS URL is derived dynamically from the token's iss claim — no hardcoded issuer needed.
  • Email extracted from X-Clerk-User-Email header or clerk_user_email cookie fallback.

Legacy Auth — HMAC Cookie

  • Cookie name: admin_session
  • Signed with ADMIN_PASSWORD secret via HMAC-SHA256.
  • Still accepted for backward compatibility but not issued to new sessions.

Access Rules

ResourceRule
/adminValid Clerk JWT + email in admin allowlist + role = 'admin' in tenant_users
/appValid Clerk JWT + any tenant membership
API routes with auth_required=TrueValid Clerk JWT (checked by middleware)
/handoverPublic — no auth
Admin on dash subdomain (prod)Redirect to admin.promostack.io/admin

Host-Aware Routing

core/host_context.py inspects the Host header to determine HostType.ADMIN or HostType.DASH and adjusts redirects accordingly. In development both subdomains map to the same instance.

6. Domain Modules

Each domain under domains/ follows the same structure:

  • handlers.py — HTTP request handlers (thin, delegate to service/repo)
  • repo.py — All database queries for this domain
  • service.py — Business logic (if needed)

Core Services

TenantRuntime (core/runtime.py)

Container for all tenant-scoped operations. Ensures every service call is isolated to a single tenant. Provides lazy-loaded access to signal engine, AI engine, and Telegram client.

BotCredentialResolver (core/bot_credentials.py)

Single source of truth for bot tokens. Raises BotNotConfiguredError with a user-actionable message if a bot is not configured. Callers should never hard-code tokens.

resolver = BotCredentialResolver(tenant_id='entrylab')
creds = resolver.get_bot_credentials('signal_bot')
# → {'token': '...', 'channel_id': '...', 'username': '...'}

Leader Election (core/leader.py)

Uses pg_try_advisory_lock(). Only the lock-holder runs background workers. Other instances poll every 10 seconds for failover.

Connections Domain (domains/connections/)

Manages tenant_bot_connections. On save: validates token via Telegram getMe, stores credentials, registers webhook with unique secret. On server startup, core/bootstrap.py re-registers all webhooks from the table.

Bot RolePurpose
signal_botPosts forex signals to VIP + FREE channels
message_bot(Legacy) Journey DMs via bot label

The generic webhook handler at /api/bot-webhook/<secret> handles: journey triggers, journey replies, AND channel join tracking (chat_member events for invite link matching).

8. Journeys System (Support Chat)

Event-triggered, multi-step conversational flows for Telegram bots. Triggered via deep links (t.me/bot?start=code) or direct keyword messages.

Step Types

TypeBehaviour
messageSend message → advance immediately
questionSend message → wait for user reply
delaySchedule next step in the future
wait_for_replyPause until reply or inactivity timeout

Journey Lifecycle States

Draft — Editable, not triggering.   Active — Live, triggers for new users.   Stopped — No new triggers; in-progress users finish.

Production Reliability Features

  • Per-user async locking — LRU-evicting, max 1000 locks. Prevents race conditions.
  • Message deduplication — journey_inbound_dedupe table prevents double-processing on webhook retries.
  • Session auto-heal — Broken sessions are marked and fall through to DM triggers.
  • Inactivity auto-complete — Configurable inactivity_timeout_days per journey, enforced in both listener and scheduler.
  • Stale job guard — Scheduler validates session status + step match before sending.
  • Journey priority — priority_int column; highest priority wins on trigger match.
  • Journey locking — is_locked = true prevents edits; duplicate endpoint clones the journey+steps+triggers.
  • Inline step delays — Delays ≤15 s executed inline (exact timing); >15 s deferred to scheduler.

Start Delay

Configurable start_delay_seconds (0–180 min). If 0, step 1 executes immediately on trigger. If >0, step 1 is scheduled via the scheduler after the configured delay. Single code path for all delays.

Await-Reply Keyword Branching

Each wait_for_reply step supports: branch_keyword (matched case-insensitively), branch_true_step_id (matched), branch_false_step_id (unmatched). Allows simple yes/no flows.

Placeholder Support

{first_name} in any message or question step text is replaced with the user's Telegram first name at execution time (stored as _first_name in session answers JSON at journey start).

9. Hype Bot (Hypechat)

AI-powered motivational messaging system for the FREE Telegram channel. Posts scheduled content to nurture free subscribers and drive VIP upgrades. Domain: domains/hypechat/.

Concepts

  • Prompts — Named AI system prompts defining the message angle/style (e.g. "Daily motivation", "Urgency"). Admin creates and names these in the UI.
  • Flows — Named sequences of steps triggered after an event (e.g. after cross promo TP1). Flows chain together via trigger_after_flow_id.
  • Steps — Individual actions within a flow, executed in sequence with per-step delays.

Step Types

TypeIconWhat it does
reforward↩Forwards a saved message from VIP to FREE. Presets: best_tp, daily_recap, weekly_recap, signal.
cta🔗Posts a call-to-action message with the VIP upgrade URL (UTM params auto-appended).
message💬Posts a plain-text message (no AI).
ai_hype✨Posts an AI-generated message using the flow's prompt + live pip performance context.

Flow Chaining

Every flow has trigger_after_flow_id (NULL = root, triggered by Cross Promo) and trigger_delay_minutes. Root flows fire from cross promo; child flows dispatch recursively after their delay. Circular chains are protected via a _visited set.

Signal Context Injection

For ai_hype and reforward steps, the live pip performance context (today + 7-day) is injected into the AI prompt automatically.

Legacy Flows

Flows with no steps fall back to _execute_flow_legacy() in service.py — backward-compatible with flows created before the step sequencer was built.

AI Message Generation

Uses gpt-4o-mini with a fixed EntryLab system prompt. For ai_hype steps, all messages in a flow's AI steps are pre-generated as a batch for narrative arc coherence before any are posted.

Admin UI

The Hype Bot admin is at #forex/hype-bot in admin.html, powered by assets/js/hypebot.js. The flow editor uses a Klaviyo-style full-page inline editor — clicking a flow replaces the page content with an editor where "+" and "✏" expand inline forms directly in the pipeline. No nested modals.

10. Cross Promo Automation

Automated system for cross-promoting VIP signals in the FREE channel. Domain: domains/crosspromo/.

Morning Sequence (UTC)

Default TimeAction
06:42Post gold morning news (Alpha Vantage)
07:12Forward weekly recap (if available) from VIP → FREE
07:51Post VIP teaser message
20:00End-of-day pip brag

TP-Triggered Lifecycle

TP1 hit → forward signal + TP1 update to FREE
         → start 30-min countdown
TP2 hit → forward TP2, reset countdown
TP3 hit → forward TP3, finish cross promo immediately
           → trigger Hype Bot flow chain

Countdown expires OR SL after TP1 → finish cross promo → trigger Hype Bot

Job Queue

Jobs stored in crosspromo_jobs. Claimed atomically with FOR UPDATE SKIP LOCKED. Job dedupe uses DELETE (not UPDATE) to free slots for rescheduling. Old CTA job removed — Hype Bot flow chain replaces it.

Admin UI

Cross Promo settings are embedded within the Signal Bot connection card on the Connections page (#forex/connections). No standalone page.

11. Telegram Infrastructure

Bot API (python-telegram-bot)

  • Uses AIORateLimiter to stay within Telegram's 30 msg/s global limit.
  • Short-lived telegram.Bot instances with a 60 s TTL cache for bot connections.
  • Explicit cache invalidation when bot tokens change.
  • Webhook URLs registered at startup from tenant_bot_connections table.
  • Single-process webhook architecture (no long polling).

Telethon User Client

Used for Journey messaging without the bot label (appears as a user sending DMs). Features:

  • Singleton client per tenant.
  • Rate limiting: soft 1200/hard 1500 daily sends. get_level() for scheduler pacing.
  • Session persistence across restarts.
  • Handles private messages for journey reply detection.

Conversion Tracking

Full customer lifecycle tracking via unique Telegram invite links:

  1. Landing page captures email + UTMs → POST /api/telegram/grant-access with amountPaid=0
  2. Server generates a single-use FREE channel invite link (72 h expiry, member_limit=1)
  3. User joins FREE channel → webhook matches invite link to email (bridges email ↔ Telegram identity)
  4. User pays for VIP → Stripe webhook detects free→VIP conversion, sets is_converted=TRUE, calculates conversion_days

Fallback: configurable public link (free_channel_link on tenant_bot_connections). Retry logic: 3 attempts with 1 s delay.

UTM Auto-append

Hype Bot CTA URLs have UTM params auto-appended: utm_source=telegram&utm_medium=free_channel&utm_campaign=hype_bot.

13. Deployment & Operations

DigitalOcean App Platform

Production is configured in .do/app.yaml. The app runs as a single service with the command python3 server.py. Secrets are marked type: SECRET, scope: RUN_TIME.

Startup Sequence

  1. server.py starts MyHTTPRequestHandler on configured port.
  2. db.py initialises connection pool and runs CREATE TABLE IF NOT EXISTS for all tables.
  3. core/bootstrap.py re-registers all Telegram webhooks from tenant_bot_connections table.
  4. Leader election begins. Winner starts all background workers.

Logging

All modules use core/logging.get_logger(__name__). Log level controlled by LOG_LEVEL env var. Logs go to stdout (captured by DigitalOcean App Platform log stream).

Adding a New Route

  1. Add a Route(...) to the appropriate list in api/routes.py.
  2. Add the handler function to the appropriate domain handlers.py.
  3. Add def handle_xxx(self): domain.handle_xxx(self) to MyHTTPRequestHandler in server.py.
  4. Validate at startup — server logs an error if any route's handler method is missing.

Planned: Multi-Tenant Forex Orchestrator

Full design in docs/MULTI_TENANT_FOREX_ROADMAP.md. Summary:

  1. Tenant Discovery — query tenant_bot_connections for all signal_bot rows.
  2. TenantOrchestrator — spawns one ForexSchedulerRunner per active tenant as an asyncio task.
  3. Health monitoring — heartbeats every minute; restart unhealthy schedulers after 5 min silence.
  4. Bootstrap integration — switch core/bootstrap.py to use orchestrator when TENANT_ID is unset.

Estimated effort: ~4 hours.

Planned: Bot Hub Marketplace

Multi-tenant SaaS marketplace where independent operators can plug in their own Telegram bots. Architecture is already multi-tenant — the main work is the tenant onboarding wizard, billing isolation, and UI whitelabelling.
